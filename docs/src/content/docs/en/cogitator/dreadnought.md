**Module:** `stc.nixosModules.cogitator-dreadnought`

A Dreadnought is an armored walker housing a venerated but mortally wounded Space Marine—the Ancient awakened from the Sarcophagus and thrust into battle, fusing flesh with machine, memory with steel. This cogitator is the awakening ritual: it takes an AMI (the sealed Sarcophagus) and resurrects it as a running EC2 instance (the Dreadnought). The instance is now operational, armed, and bound to the hardened substrate of your infrastructure, awaiting its orders on the battlefield.

Operating profile for NixOS EC2 instances deployed from a
[sarcophagus-aws](/stc/en/cogitator/sarcophagus-aws/) AMI. Activates ZFS,
networking, impermanence, AWS platform support, and the full hardening suite
on the running system.

No disk layout or image builder is included — the AMI already laid the disk.
The ZFS filesystem layout and GRUB boot loader configuration are owned by this
cogitator (derived from `poolName` and `ebsDisk`), so the consumer's
`hardware-configuration.nix` only needs to provide the EC2-specific kernel
modules generated by `nixos-generate-config`.

Add other cogitators on top (`cogitator-docker-server`, `cogitator-vm`, etc.)
to build the service profile you need.

## What It Composes

| Component | What it does |
|-----------|-------------|
| `relics-zfs` | ZFS-compatible kernel, auto-scrub, TRIM, ZED |
| `relics-networking` | DHCP, Quad9 DNS |
| `relics-impermanence` | Root rollback to `@blank` on every boot |
| `relics-aws` | NVMe/ENA drivers, NTP, serial console, EBS pool expansion |
| `cogitator-hardening` | Kernel + network + filesystem + SSH hardening |

`impermanence.nixosModules.impermanence` is injected automatically.

## Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `stc.cogitator.dreadnought.enable` | bool | `false` | Enable this cogitator |
| `stc.cogitator.dreadnought.poolName` | str | `"vmpool"` | ZFS pool name — must match the pool created by sarcophagus-aws |
| `stc.cogitator.dreadnought.ebsDisk` | str | `"nvme0n1"` | EBS device name (without `/dev/`). Use `xvda` for Xen instances. |
| `stc.cogitator.dreadnought.ebsPartition` | int | `3` | ZFS pool partition number on the EBS disk |
| `stc.cogitator.dreadnought.impermanence.extraDirectories` | list of str | `[]` | Additional directories to persist under `/persist` |
| `stc.cogitator.dreadnought.impermanence.extraFiles` | list of str | `[]` | Additional files to persist under `/persist` |

## Usage Example

```nix
# flake.nix
{
  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixos-unstable";
    stc = {
      url = "github:gfriloux/stc";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { nixpkgs, stc, ... }: {
    nixosConfigurations.my-server = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      modules = [
        stc.nixosModules.cogitator-dreadnought

        # Add service profiles on top
        stc.nixosModules.cogitator-docker-server  # example

        # Hardware configuration generated from the running instance
        ./hardware-configuration.nix

        ({ ... }: {
          stc.cogitator.dreadnought = {
            enable = true;
            poolName = "vmpool"; # must match the AMI's pool name
            impermanence.extraDirectories = [
              "/var/db/sudo/lectured"
              "/var/lib/docker"
            ];
          };
          # stc.cogitator.docker-server.enable = true; # if using that cogitator

          networking.hostName = "my-server";
          networking.hostId = "abcd1234"; # must match the value set in the AMI
          users.mutableUsers = false;
          system.stateVersion = "24.11";
        })
      ];
    };
  };
}
```

## Hardware Configuration

The filesystem layout and GRUB boot loader are fully managed by this cogitator.
The consumer's `hardware-configuration.nix` only needs the kernel modules
section — run on the running instance:

```bash
nixos-generate-config --show-hardware-config
```

Copy only the `boot.initrd.availableKernelModules` and `boot.kernelModules`
lines into your `hardware-configuration.nix`. The filesystem declarations and
boot loader device are already set by the cogitator and do not need to be
repeated.
